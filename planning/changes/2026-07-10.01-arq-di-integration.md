# Design: modern-di integration for arq

**Status:** design, pending maintainer approval
**Target:** new repo `modern-python/modern-di-arq`, package `modern_di_arq`

## Summary

Ship the official [modern-di](https://modern-di.modern-python.org) integration
for [arq](https://arq-docs.helpmanual.io) (async Redis task/job queue),
following the core repo's
[Writing an integration](../../../modern-di/docs/integrations/writing-integrations.md)
contract along the **decorator path** (arq tasks are plain coroutines with no
native DI seam — the async analog of `modern-di-celery`/`-typer`). One module
(`modern_di_arq/main.py`) exposes `setup_di`, `fetch_di_container`, `FromDI`,
and `inject`. `setup_di` wires the root container's lifecycle to arq's
`on_startup`/`on_shutdown` and seeds it into arq's `ctx` dict; a per-job
`Scope.REQUEST` child is built in `on_job_start` and closed in `on_job_end`;
`@inject` resolves `FromDI` params from that child.

## Motivation

arq is a widely-used async Redis job queue, common in FastAPI stacks. It is the
roadmap's top "Next up" integration. Dishka has an arq integration; modern-di
has none. arq's async model is a clean fit for modern-di.

## Verified facts (arq 0.28 + modern-di 2.27, spiked end-to-end via a real burst Worker on Redis)

Every non-obvious decision below was proven against installed arq, not inferred:

- **arq runs a task as `function.coroutine(ctx, *args, **kwargs)`** — the `ctx`
  dict is passed as argument 0, followed by the enqueued args. arq does **no
  signature binding** of the task (the only `inspect.signature` call in the
  worker is on the `Worker` class itself, not on tasks). So **no signature
  rewrite is needed** (unlike Celery/aiogram) and **`functools.wraps` is safe**
  — arq reads only `coroutine.__qualname__` for the default job name, which
  `functools.wraps` preserves, and never unwraps to defeat anything.
- **`func()` calls `inspect.iscoroutinefunction(coroutine_)` and raises
  `RuntimeError` if false** — so the `@inject` wrapper **must be `async def`**.
  It is; `functools.wraps` does not change a genuine `async def` wrapper's
  coroutine-function status.
- **`ctx` is one dict object across a job's lifecycle.** arq builds
  `ctx = {**self.ctx, **job_ctx}` once per job, then calls `on_job_start(ctx)`,
  the task `coroutine(ctx, ...)`, and `on_job_end(ctx)` with that same object.
  So `on_job_start` can stash the per-job child in `ctx` and the task's
  `@inject` reads it back, and `on_job_end` closes it. Proven end-to-end.
- **`self.ctx` (worker-level) is merged into every per-job `ctx`.** Seeding the
  root container into the worker `ctx` (via `worker_settings.ctx`) at setup time
  makes it reach every job's `ctx`. Proven: a task saw the seeded container.
- **`on_startup`/`on_shutdown` own worker lifecycle**; `on_job_start`/
  `on_job_end` own per-job lifecycle. All four receive the `ctx` dict and are
  `async`. `setup_di` composes with any hooks the user already set (their hook
  still runs). Proven: the user's own `on_startup` ran alongside ours.
- **arq's `WorkerSettings` is the dominant idiom** — a class with `functions`,
  `redis_settings`, and optional hooks; run via `arq module.WorkerSettings`.
  arq also accepts a `dict` of settings (`Worker(**settings)`). `setup_di`
  supports **both** forms (attribute access for a class/object/instance, item
  access for a dict), mirroring Dishka.
- **A modern-di `Container` is created open** (`closed=False`); `.open()` only
  *reopens* after a shutdown-close. `on_startup` → `container.open()` makes a
  second worker run (test re-entry / restart) reuse the container without a
  `ContainerClosedWarning`; `on_shutdown` → `container.close_async()` runs
  APP-scoped finalizers. Dishka omits root lifecycle management; modern-di's
  contract requires it, so this integration wires it.
- **Testing needs a real Redis** — arq has no in-memory broker. A burst Worker
  (`burst=True`) against a Docker/CI `redis:8` service runs all queued jobs and
  exits; `Worker(redis_pool=...)` accepts an injected pool. Proven end-to-end.

## Design

### Connection provider (kind → scope)

**None.** arq's per-job context is a bare `dict[str, Any]` (`job_id`, `job_try`,
`enqueue_time`, `score`, `redis`) — a poor DI key, and modern-di does not expose
one for it (the same call as `modern-di-celery`/`-typer`, which have no
injectable connection object). A task that needs job metadata reads its `ctx`
argument directly.

### Container handoff (two named `ctx` keys)

arq's `ctx` dict is the framework's state store (its `app.state` analog):

- `_ROOT_CONTAINER_KEY = "modern_di_container"` — the root container, seeded into
  the worker `ctx` by `setup_di`, read by `fetch_di_container`, and used by
  `on_job_start` to build the per-job child.
- `_CHILD_CONTAINER_KEY = "modern_di_request_container"` — the per-job child,
  stashed on the per-job `ctx` by `on_job_start`, read by `inject`, closed by
  `on_job_end`.

### `setup_di(worker_settings, container) -> Container`

Argument order matches modern-di's uniform `setup_di(app, container)` (arq's
`worker_settings` is the "app"), **not** Dishka's reversed order. `setup_di`:

1. Seeds `container` into the worker `ctx` under `_ROOT_CONTAINER_KEY` (creating
   `ctx` if absent), preserving any existing `ctx` entries.
2. Wraps `on_startup`/`on_shutdown` to open/close the root container (composing
   with any existing hook).
3. Wraps `on_job_start`/`on_job_end` to build/close the per-job child (composing
   with any existing hook).

It supports both a class/object `worker_settings` (attribute access) and a
`dict` (item access). Returns the container.

```python
def setup_di(worker_settings, container):
    ctx = _get_setting(worker_settings, "ctx") or {}
    ctx[_ROOT_CONTAINER_KEY] = container
    _set_setting(worker_settings, "ctx", ctx)

    _set_setting(worker_settings, "on_startup",
                 _wrap_startup(container, _get_setting(worker_settings, "on_startup")))
    _set_setting(worker_settings, "on_shutdown",
                 _wrap_shutdown(container, _get_setting(worker_settings, "on_shutdown")))
    _set_setting(worker_settings, "on_job_start",
                 _wrap_job_start(_get_setting(worker_settings, "on_job_start")))
    _set_setting(worker_settings, "on_job_end",
                 _wrap_job_end(_get_setting(worker_settings, "on_job_end")))
    return container
```

`_get_setting`/`_set_setting` dispatch on `isinstance(worker_settings, dict)`
(item access) vs. otherwise (attribute access via `getattr`/`setattr`).
`_get_setting` returns `None` when the key/attribute is absent.

### Hook wrappers

```python
def _wrap_startup(container, existing):
    async def on_startup(ctx):
        container.open()                       # reopen for restart / test re-entry
        if existing is not None:
            await existing(ctx)
    return on_startup

def _wrap_shutdown(container, existing):
    async def on_shutdown(ctx):
        if existing is not None:
            await existing(ctx)
        await container.close_async()          # run APP-scoped finalizers
    return on_shutdown

def _wrap_job_start(existing):
    async def on_job_start(ctx):
        root = ctx[_ROOT_CONTAINER_KEY]
        ctx[_CHILD_CONTAINER_KEY] = root.build_child_container(scope=Scope.REQUEST)
        if existing is not None:
            await existing(ctx)
    return on_job_start

def _wrap_job_end(existing):
    async def on_job_end(ctx):
        if existing is not None:
            await existing(ctx)
        child = ctx.pop(_CHILD_CONTAINER_KEY, None)
        if child is not None:
            await child.close_async()          # never leak on the error path
    return on_job_end
```

Ordering rationale: user hooks run **after** ours on setup (root open, child
built) and **before** ours on teardown (child close, root close), so a user hook
always sees a live container. `on_job_end` uses `ctx.pop(..., None)` so it is a
no-op if `on_job_start` never ran (defensive; arq always pairs them).

### `fetch_di_container(ctx) -> Container`

```python
def fetch_di_container(ctx):
    return typing.cast(Container, ctx[_ROOT_CONTAINER_KEY])
```

Reads the root container back out of any per-job or worker `ctx` dict.

### `FromDI` marker + `inject` decorator

`FromDI` is an inert, frozen, slotted dataclass marker (the decorator-path
shape, identical to `modern-di-celery`). `inject` parses `Annotated[T,
FromDI(...)]` params, and — because arq passes `ctx` + enqueued args
positionally against the task's full parameter list — uses the **Celery
bind-by-name** resolution so injection is **parameter-order-insensitive**:

```python
def inject(func):
    di_params = _parse_inject_params(func)
    if not di_params:                          # nothing to inject → passthrough
        return func

    signature = inspect.signature(func)
    visible_params = [p for name, p in signature.parameters.items() if name not in di_params]
    visible_signature = signature.replace(parameters=visible_params)

    @functools.wraps(func)                      # safe for arq; preserves __qualname__ for the job name
    async def wrapper(*args, **kwargs):
        ctx = args[0]                           # arq always passes ctx first
        child = ctx[_CHILD_CONTAINER_KEY]       # built by on_job_start
        resolved = {name: child.resolve_dependency(marker.dependency)
                    for name, marker in di_params.items()}
        bound = visible_signature.bind(*args, **kwargs)
        bound.apply_defaults()
        return await func(**bound.arguments, **resolved)

    return wrapper
```

`visible_signature` (the task minus its DI params, still including `ctx` and the
real args) binds the incoming positional `ctx` + enqueued args by name; resolved
DI params are added by name. `resolve_dependency` dispatches provider-vs-type.
Unlike Celery, the wrapper does **not** set `wrapper.__signature__` (arq never
reads a task signature) and **does** use `functools.wraps` (arq reads
`__qualname__` for the default job name; `functools.wraps` preserves it).

### Public API (`__all__`)

`setup_di`, `fetch_di_container`, `FromDI`, `inject`.

## Non-goals

- **auto-inject.** arq's `WorkerSettings.functions` list is heterogeneous (plain
  coroutines, `arq.func(...)` wrappers, import strings), so blanket wrapping is
  fragile. v1 ships explicit `@inject` per task only. (Deferrable if demanded.)
- **A connection `ContextProvider`.** arq's `ctx` is a bare `dict`; no injectable
  connection object (matches Celery/Typer).
- **Sync resolution / sync finalizers only** — arq is async; the integration is
  fully async (`close_async`).
- **A signature-rewriting `inject`** (Celery/aiogram needed one; arq does not —
  it never binds the task signature).
- Any runtime dependency beyond `arq` and `modern-di`; a separate docs site.
- **Cron jobs** get no special handling — an `@inject`-decorated coroutine works
  as a `cron` job too (arq calls it the same `coroutine(ctx, ...)` way).

## Testing

Real Redis via `docker-compose` (a `redis:8` service) locally and a GitHub
Actions `redis:8` service in CI, mirroring
[`faststream-redis-timers`](../../../faststream-redis-timers): a `redis_client`/
pool fixture reads `REDIS_URL` and `pytest.skip`s if Redis is unreachable, with
uuid-suffixed queue names for isolation. Tests drive a **real burst Worker**
(`burst=True`, `handle_signals=False`, injected `redis_pool`) so the full arq
path (hook composition, `ctx` merge, task dispatch) is exercised, not just direct
coroutine calls. 100%-coverage gate (`just test-ci`).

- `dependencies.py` — a `Group` with APP- and REQUEST-scoped `Factory`
  providers, including one with an async finalizer (to prove per-job child
  teardown) and one APP-scoped finalizer (to prove root teardown on shutdown).
- `test_lifespan.py` — running a burst Worker opens the root container and closes
  it on shutdown (APP finalizer runs); running the Worker a **second time** on
  the same container reopens it with **no `ContainerClosedWarning`** (restart
  safety); `fetch_di_container(ctx)` returns the same root.
- `test_jobs.py` — an `@inject` task resolves APP-scoped and REQUEST-scoped
  values through a real enqueued+executed job; a task with **no** `FromDI` is a
  passthrough; injection is **order-insensitive** (a `FromDI` param before a real
  enqueued arg resolves correctly); the per-job child's async finalizer runs
  after the job (`on_job_end` closed it).
- `test_setup.py` — `setup_di` works for both a **class** `WorkerSettings` and a
  **dict** settings form; it **composes** with a user-supplied `on_startup` and
  `on_job_start` (both the user's hook and ours run).

## Risk

- **R1 — arq is pre-1.0 (0.28 latest); API drift.** `ctx` merge semantics,
  `on_startup`/`on_shutdown`/`on_job_start`/`on_job_end` hooks, `func()`'s
  `iscoroutinefunction` check + `__qualname__` naming, and `Worker(redis_pool=)`
  are relied on. *Mitigation:* pin `arq>=0.25,<1`. The implementation step
  installs the low bound (`arq==0.25`) on Python 3.10 and runs the suite to
  confirm those APIs exist there; **if one landed later, raise the floor to the
  lowest green version and report it** (the aiogram 3.0→3.2 / Flask 2→3 pattern).
- **R2 — `on_job_start` builds a child for every job**, even a task with no
  `FromDI`. Negligible overhead (one child container per job), and correct;
  `@inject`'s passthrough branch means an undecorated task simply ignores the
  child. Not optimized in v1.
- **R3 — `functools.wraps` sets `__wrapped__`.** arq's `func()` calls
  `inspect.iscoroutinefunction`, which does **not** unwrap in Python 3.10+, and a
  genuine `async def` wrapper reports as a coroutine function regardless.
  Verified. (The aiogram trap — where the framework *did* unwrap — does not
  apply to arq.)
